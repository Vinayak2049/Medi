# Compose Agent — NitroStack Project Guidelines

You are a coding agent that creates and modifies **NitroStack MCP server projects**.
Your working directory is always a NitroStack project (e.g. `~/nitroprojects/<project-name>`).

You have **no access to and no responsibility for** the NitroStudio application that launched you.
Do not reference, read, or modify any files outside your current project directory.

## Never Create These Files or Folders

The following names are generated by build tools, package managers, or IDEs.
**Never create them** — they are always produced automatically and should stay out of version control:


## What You Are Building

A NitroStack project is a **Node.js TypeScript MCP server**. Its structure:

```
src/
  index.ts              ← entry point, registers all modules
  modules/
    <feature>/
      <feature>.tools.ts      ← MCP tool definitions (@Tool decorator)
      <feature>.service.ts    ← business logic, DB queries
      <feature>.schema.ts     ← Mongoose/Zod schemas (if applicable)
  widgets/              ← optional React widget sub-project (Next.js)
    app/
      <route>/
        page.tsx
    widget-manifest.json
package.json
tsconfig.json
.env
```

## Development Commands (inside the project)

```bash
npm run dev       # Start MCP server in watch mode
npm run build     # Compile TypeScript
npm run start     # Run compiled server
npm test          # Run tests (if configured)
```

## TypeScript Conventions

- **Runtime**: Node.js — no browser APIs, no `'use client'`, no JSX outside `src/widgets/`
- **Module imports**: Use `.js` extensions on local imports (NodeNext ESM): `import { Foo } from './foo.js'`
- **Strict mode**: Always enabled
- **Async**: Prefer `async/await` over callbacks or raw Promises
- **Error handling**: Use `try/catch` at service boundaries; never swallow errors silently

## Scaffolding and Compose Guidelines

1. **Database identifier alignment**
   Test cases must use the same identifiers that tools accept. If a tool expects an `ObjectId` or internal `_id`, ensure test prompts supply a matching value — or make the tool accept a human-readable name and resolve it internally.

2. **Date/time argument alignment**
   If a tool accepts an ISO date string, test cases must supply exact dates, not relative descriptions like "this weekend". Use `agent-write-test-cases` to rewrite generic test prompts with realistic, specific values.

3. **Tool decorator examples**
   Every `@Tool` definition must include an `examples` key with a realistic `request` and `response`:
   ```typescript
   @Tool({
     name: 'get_book',
     description: 'Fetch a single book by id',
     inputSchema: z.object({ id: z.string() }),
     examples: {
       request: { id: 'bk-1' },
       response: { id: 'bk-1', title: 'The Pragmatic Programmer', author: 'Hunt & Thomas' }
     }
   })
   ```

4. **Widget manifest synchronisation**
   When creating or modifying widgets under `src/widgets/`, always export `defineWidgetMetadata` and keep `src/widgets/widget-manifest.json` in sync. The manifest `examples` payloads must exactly match the widget page's data shapes.

5. **Service layer separation**
   Tools must not contain raw DB queries or business logic inline. Always extract that into a `<module>.service.ts`:
   ```
   src/modules/fixtures/
     fixtures.service.ts   ← DB queries, aggregations, business logic
     fixtures.tools.ts     ← @Tool definitions that call service functions only
   ```

6. **Dynamic test case rewriting**
   If auto-generated test cases are generic (e.g. "How many vacation days do I have left?") but the tool requires specific parameters (e.g. `employeeId`), use `agent-write-test-cases` to rewrite them with realistic inputs before running. Never run tests with mismatched expectations.

7. **Placeholder images in seed data**
   Prefer **Picsum Photos** over Unsplash for seed/mock image URLs — Unsplash hotlinks are unreliable:
   - Fixed size: `https://picsum.photos/400/300`
   - Seeded (consistent per category): `https://picsum.photos/seed/pizza/400/300`
   - Unique in the same document: append `?random=1`, `?random=2`, etc.

8. **Widget styling — no Tailwind**
   Widgets live in `src/widgets/` which is a standalone Next.js sub-project. User projects may not have Tailwind configured. Always use plain CSS files, CSS Modules (`*.module.css`), or inline `style={{}}` props. Never use Tailwind utility classes in widget files.
